# Main 功能交付质量标准

本文定义 TaskbarLyrics main 分支对“功能已交付”的质量标准。它不是功能数量对比，也不是要求 Light 合并 main 的 App 层实现；它用于说明 main 如何判断一个功能是否真的完成，以及 Light 在每个 main 发布节点应如何对齐已有功能的交付质量。

main 仍以 WPF + WebView2 的视觉体验方向为主。TaskbarLyrics 的核心场景是“任务栏歌词”：显示区域小、用户注意力有限、窗口需要稳定贴合任务栏，功能设计必须服务这个场景。只完成入口、配置字段或基础 UI，不代表达到 main 的交付标准。

## 1. 文档目的

- 统一 main 对歌词检索、歌词显示、设置页三块功能的交付口径。
- 区分“功能能跑”和“功能可发布”之间的差距。
- 为 PR 评审、Release 检查、Light 发布节点质量对齐提供共同标准。
- 避免为了追功能数量而引入伪需求、尚未打磨成熟的 UI 或破坏任务栏显示体验的复杂配置。

## 2. 分支关系与对齐方式

| 分支 | 定位 | 对齐方式 |
| --- | --- | --- |
| main | 产品质量基准，继续以视觉体验和任务栏显示完成度为主。 | 通用功能和核心体验优先在 main 内按本标准打磨。 |
| light | 轻量版分支，保留原生 WPF、低进程数、低内存和较小发布体积方向。 | 不需要合并 main 的 App 层实现，但应在 main 每个 Release 节点对照本标准补齐已有功能质量。 |

Light 的对齐目标不是把 main 的 WebView2 实现搬过去，也不是要求原生 WPF 逐像素复刻 WebView2 的视觉效果，而是把自己已经实现的同类能力打磨到可交付状态。例如：歌词显示可以继续用原生 WPF，并允许在动效、材质和细节质感上按 WPF 能力适当放宽；但长歌词、省略、动画节奏、窗口边界、任务栏可用区域、设置持久化和异常路径仍需要达到同等级的体验目标。

## 3. main 的交付原则

| 原则 | 要求 |
| --- | --- |
| 视觉优先 | main 的歌词窗和设置页以视觉质感、动画自然度、布局稳定性为主要方向。性能优化不能以明显视觉降级为代价。Light 对齐时不要求 WPF 一比一复刻 WebView2，但应尽量保留 main 的可读性、克制感、动效节奏和布局稳定性。 |
| 服务任务栏场景 | 功能必须适配任务栏的小尺寸、低打扰、贴边显示和快速扫读场景。超出任务栏区域的重组件、高密度信息堆叠、桌面播放器式面板不应作为默认方向。 |
| 完整而克制 | 功能应覆盖正常路径、失败路径和边界路径，但设置项不应为了“看起来功能多”而堆叠。 |
| 状态一致 | 歌词窗、设置页、托盘菜单、缓存、启动行为、播放器状态必须一致，不能出现一个入口开了另一个入口不生效。 |
| 可诊断 | 检索失败、SMTC 异常、封面缺失、更新失败等问题应能通过日志或诊断入口定位。（这一点main也仍在完善）） |

## 4. 歌词检索

歌词检索主要由 `TaskbarLyrics.Core` 承担，Light 也复用这一层能力。因此 Light 与 main 在歌词检索上的关系应尽量简单：Light 不需要另起一套检索逻辑，也不需要把检索能力作为分支差异重点；如果在使用中发现 Core 的检索、匹配、缓存、翻译解析或本地歌词策略存在可优化点，应拆成面向 main 的独立 PR 回流。

歌词检索相关改动进入 main 前，重点看三件事：

- 是否提升命中质量、失败回退、缓存一致性或可诊断性。
- 是否能在失败、超时、错配、缓存命中等边界下保持歌词窗和设置页状态稳定。

Light 在发布节点只需要确认自己正确接入 Core，并跟进 main 已合入的 Core 修复。主要对齐工作不应放在检索能力本身，而应放在歌词显示和设置页这些直接影响用户感知的 App 层体验。

## 5. 歌词显示

歌词显示是 main 的核心体验，也是 Light 最需要对齐交付质量的地方。main 的标准不是“能把文字画出来”，而是文字、动画、封面、频谱、任务栏区域和窗口生命周期共同达到稳定、顺滑、可读。考虑到 Light 采用原生 WPF，视觉评审不要求它与 WebView2 版本完全一致；对齐重点应放在体验意图上，包括可读性、信息密度、动画不打扰、窗口不漂移和任务栏区域内的整体克制。

| 功能 | main 当前实现 | 交付质量要求 | 尚需打磨的表现 | 验收方式 |
| --- | --- | --- | --- | --- |
| 双行歌词 | WebView2 歌词窗支持双行歌词和平滑切换。 | 当前行、下一行、翻译行层级清楚；长文本省略自然；切歌和暂停时不跳动。 | 文本重叠、突然换行撑高窗口、长歌词被硬裁切、切换时闪烁。 | 使用短句、长句、中英混排、含翻译歌词验证。 |
| 歌词切换动画 | main 有平滑切换动画。 | 动画要服务扫读，不能抢注意力；帧率稳定；暂停、快进、切歌时状态能重置。 | 动画炫但突兀；快速切歌残留上一句；低性能机器明显卡顿。 | 快速切歌、拖动播放进度、暂停恢复验证。 |
| 翻译显示 | 主要与core能力层相关。 | 在core补齐音源翻译能力前不做要求。 |  |  |
| 封面显示 | main 已有 SMTC 封面和本地封面能力。 | 封面应增强识别，不应喧宾夺主；加载失败、切换封面、无封面时要自然过渡。 | 封面加载闪烁；无封面留空洞；封面过大挤压歌词。 | 切换有封面、无封面、本地封面、网络封面歌曲验证。 |
| 频谱显示 | main仍在探索最佳效果参数。 | 暂不做交付要求。 |  |  |
| 字体、颜色、字重 | main 支持字体、前景色和字重设置。 | 默认值应足够好看；颜色与背景有可读对比；字体变更后不破坏布局。 | 暴露大量字体选项但默认难看；颜色导致文字不可读；字重切换后文本抖动。 | 使用浅色/深色背景、不同字体、不同字重验证。 |
| 背景、边框、阴影 | main 支持背景、边框、文字阴影等外观项。 | 视觉效果要服务可读性；透明度和阴影不能造成脏边；关闭后应干净。 | 边框/阴影像调试效果；背景遮挡任务栏；关闭后仍残留样式。 | 在浅色、深色、透明任务栏下验证。 |
| 窗口宽度、锚点、偏移 | main 支持窗口宽度、水平锚点和 X/Y 偏移。 | 窗口应贴合任务栏区域；多屏、DPI、任务栏位置变化后不漂移；偏移设置可恢复。 | 窗口跑出任务栏；高 DPI 下位置错误；切换屏幕后不更新。 | 多屏、不同缩放、任务栏左右/底部位置、重启后验证。 |

歌词显示相关需求如果会破坏任务栏歌词的核心形态，应默认暂缓。例如：

- 大面积溢出任务栏区域的封面、歌词卡片、控制面板。
- 同时堆叠封面、歌名、歌手、双行歌词、翻译、频谱和状态徽标，导致小尺寸区域不可读。
- 把任务栏歌词做成桌面悬浮播放器，偏离低打扰、快速扫读的初衷。
- 为了展示功能数量而增加复杂布局，但没有处理长文本、省略、DPI、窗口边界和动画状态。

## 6. 设置页（main 仍处于建设中期）

设置页不是控件堆叠区。main 的设置页仍在建设中，本节不把它写成过细的逐项验收清单，而是先明确建设方向和质量边界：设置页要帮助用户把任务栏歌词调到舒服、稳定、可预期的状态，而不是把所有内部参数都暴露出来。一个设置项是否成立，取决于它是否服务任务栏歌词场景，以及用户是否能理解它解决什么真实问题。

| 方向 | main 当前状态 | 建设期质量边界 |
| --- | --- | --- |
| 播放源与歌词来源 | 已支持播放器启用开关、识别顺序和本地歌词目录。后续可以继续补齐音源级设置。 | 设置项应解释“影响什么结果”，避免用户只看到一组内部开关。排序、禁用、本地目录变更应最终反映到实际检索结果。 |
| 显示与外观 | 已有翻译、频谱、字体、颜色、背景、边框、阴影等外观相关设置。 | 外观设置必须服务任务栏小空间的可读性。不要为了展示能力而鼓励大封面、大面板、高密度信息或复杂装饰堆叠。 |
| 窗口布局 | 已支持宽度、锚点、偏移等基础布局设置。 | 布局设置需要有边界意识，不能轻易把歌词窗调出任务栏可视区域；后续如果增加高度、自动尺寸等能力，应优先保证多屏、DPI 和任务栏位置变化下稳定。 |
| 维护与诊断 | 已有缓存清理、SMTC 监视器、更新检查等维护能力。 | 维护类入口应集中、低风险、反馈明确。诊断能力可以逐步完善，但不应把原始内部字段直接堆给普通用户。 |
| 默认值与恢复路径 | 部分设置已有默认值，设置页整体仍在迭代。 | 默认值应体现 main 的审美和任务栏场景判断；高风险或容易调坏体验的设置应提供范围限制、恢复默认或清晰说明。 |

设置项进入 main 前，应先通过伪需求检查：

- 它解决的是任务栏歌词的真实使用问题，还是只是因为实现了底层能力所以顺手暴露。
- 它会不会鼓励用户把歌词窗调成溢出任务栏区域的组件。
- 它会不会在小任务栏高度里堆积高密度信息，降低快速扫读能力。
- 它是否有合理默认值、范围、说明、即时反馈和恢复路径。
- 它是否与已有托盘菜单、启动设置、显示状态和缓存状态保持一致。

不建议作为 main 默认方向的设置项：

- 以大封面、大信息面板、大面积背景材质为核心的桌面播放器式布局。
- 需要用户理解大量内部参数才能调好的显示模式。
- 为了覆盖边缘偏好而显著增加设置页复杂度的低频选项。
- 没有边界限制，容易让窗口离开任务栏可视区域的尺寸或偏移设置。

## 7. PR 与 Release 验收口径

| 状态 | 含义 | 处理建议 |
| --- | --- | --- |
| 可合入 main | 功能符合 main 技术方向，满足本文质量要求，且不破坏现有体验。 | 可进入 main，必要时补文档和截图。 |
| 可合入 light | 功能符合 Light 技术方向，但尚未达到 main 的视觉或交付标准，或只适合轻量版。 | 可进入 light，但 Release 文案应谨慎标注成熟度。 |
| 实验功能 | 功能方向可能有价值，但交互、边界、稳定性、诊断不足。 | 留在开发分支，不进入正式 Release 说明。 |
| 需要继续打磨 | 功能入口已存在，但明显缺少状态一致性、视觉细调、边界处理或错误反馈。 | 不建议作为正式能力发布。 |
| 不接受 | 破坏任务栏歌词场景、引入发布链路风险、复制大量难维护 App 层，或与 main 方向冲突。 | 退回或要求拆分为可复用的小 PR。 |

评审时应避免只看功能覆盖面。一个功能如果没有处理边界、没有打磨视觉、没有接入现有状态流，哪怕 README 中描述得很完整，也应视为尚未达到正式发布标准。

## 8. Light 发布节点质量对齐流程

Light 不需要合并 main 分支，也不需要复制 main 的 WebView2 App 层。建议 Light 在每个 main Release 节点执行一次质量对齐：

1. 以 main 最新 Release 为质量基准。
2. 按歌词检索、歌词显示、设置页列出 Light 已有同类功能。
3. 对每项功能标记状态：
   - 达到 main 质量。
   - 功能可用但仍需打磨。
   - 尚未达到正式发布标准，不进入正式说明。
   - 实验功能，仅保留在开发分支。
4. 对“功能可用但仍需打磨”和“尚未达到正式发布标准”的项目列出补齐项，重点检查任务栏区域、长文本、省略、DPI、启动状态、设置持久化、错误反馈和资源释放。
5. 补齐后再进入 Light Release changelog。

Light 的歌词检索因为主要共享 main Core，只需要确保 Core 更新、设置入口和诊断行为没有偏离即可。Light 的主要对齐工作应放在歌词显示和设置页：这些区域决定用户感知，也最容易出现“功能已有基础，但仍需要产品化打磨”的状态。

## 9. 当前建议重点

main 后续评审和自查时，建议优先关注：

- 歌词显示是否稳定服务任务栏小空间，而不是扩展成复杂悬浮面板。
- 设置页是否按真实使用场景组织，而不是按实现字段堆控件。
- 视觉功能是否经过长歌词、翻译、封面缺失、频谱、DPI、多屏、任务栏位置变化的反复细调。
- 更新、安装、发布链路是否只指向可信官方源，并能被 review。
- Light 独有功能是否在尊重 WPF 技术边界的前提下，经过同等级体验打磨后再进入正式 Release 描述。

功能越多，越需要明确交付标准。TaskbarLyrics main 的质量基准应坚持：功能可以逐步增加，但每个正式发布的功能都要像真实日常使用过一样可靠、克制、好看。
